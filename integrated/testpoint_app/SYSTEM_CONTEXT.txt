Final Project G7 - System Context Overview
============================================

Project Root: Final-Project-G7
Last Updated: May 5, 2026

This project is a Flask-based online examination and academic user management system branded as TestPoint. It is implemented in Python with MySQL as the database backend and is organized using Flask Blueprints for separate Authentication, Student, Teacher, and Admin functionality.

---

1. Application Entry and Initialization
----------------------------------------

app.py
------
- Imports Flask, redirect, url_for.
- Imports create_app() from `testpoint.__init__`.
- Builds the Flask application via the factory pattern.
- Defines the root URL `/` to redirect users to the authentication login page.
- Applies `after_request` headers to disable browser caching for all responses.
- Runs the Flask development server in debug mode on port 5000 when executed directly.

`testpoint/__init__.py`
------------------------
- Loads environment variables from `testpoint/passwordDB.env` using `dotenv`.
- Reads database password and Gmail SMTP credentials from environment variables.
- Defines `db_config` with MySQL connection settings for `localhost`, `root`, and `test_point` database.
- Configures Flask-Mail using Gmail SMTP on port 465 with SSL.
- Initializes the mail extension and registers blueprints:
  - `auth` at `/` (authentication routes)
  - `admin` at `/admin`
  - `teacher` at `/teacher`
  - `student` at `/student`
- Uses a hard-coded Flask secret key: `Secret@123_key`.

This means the application is a modular, role-oriented web system using shared database connectivity and email support.

2. Authentication Module
------------------------

File: `testpoint/Auth/login.py`

Core behavior:
- Supports login for three active roles: `admin`/`super_admin`, `teacher`, and `student`.
- Supports pending registration accounts for new students and teachers.
- Handles student registration at `/register/student` and teacher registration at `/register/teacher`.
- Sends OTP emails for user verification and enforces OTP expiration.
- Implements document upload for pending users and verifies upload file type as PDF.
- Tracks registration and approval state using `pending_users.verification_status`.
- Supports password reset with an OTP workflow using an `otp_table` and per-hour rate limits.
- Uses session values to maintain role, login state, and registration flow state.

Key features:
- `generate_id(role_prefix)`: names users like `S26-0001`, `T26-0001`, `A26-0001` based on year and role.
- `generate_unique_otp()`: produces a 6-digit OTP.
- `send_otp_email()` and `send_reset_otp_email()`: compose and send HTML email content.
- Name and email validation using regex rules.
- `allowed_file()` ensures uploaded verification documents are PDFs.

Login flow:
- Checks active user credentials in `users`.
- Directs admin users to admin dashboard, students to student dashboard, teachers to teacher dashboard.
- If credentials match a pending user, the pending flow continues:
  - unverified users are redirected to OTP verify page.
  - verified users who still need document upload are redirected to upload page.
  - pending users with approval waiting status see a waiting screen.

Registration and verification:
- Pending user registration inserts records into `pending_users` and sends OTP.
- OTP verification updates `pending_users.is_otp_verified`.
- Document upload saves PDF to `testpoint/static/uploads/verifications` and sets `verification_status` to `pending_approval`.
- The app keeps pending user sessions via `pending_user_logged_in`.
- OTP resend is limited to 5 times per hour.

Password Reset Flow:
- The user submits email to `/forgot-password`.
- The app records OTPs in `otp_table` and enforces a 5-per-hour limit.
- `/verify-reset-otp` verifies the reset OTP.
- `/reset-password` allows password update after OTP validation.
- Reset state is tracked with session keys: `reset_email`, `reset_user_id`, `in_reset_flow`, and `otp_verified`.

Session helpers:
- `user_logged_in()` for student sessions.
- `teacher_logged_in()` for teacher sessions.
- `admin_logged_in()` for admin sessions.
- `pending_user_logged_in()` for pending verification accounts.

3. Student Module
-----------------

File: `testpoint/Student/student.py`

Student features:
- `student_dashboard` showing:
  - number of active enrolled classes.
  - number of completed exams.
  - available exams still open.
  - recent score trend.
  - class ranking per enrolled class.
- `profile` page with student data and password update support.
- `student_exams` list with computed availability status labels such as Available, Upcoming, Completed, or Unavailable.
- Uses the current datetime and exam duration to calculate exam availability.
- Injects active enrolled courses into the template context for all student pages.
- Enforces a lockdown mode during an active exam via `before_app_request`
  - prevents students from navigating away from exam-related endpoints when `active_exam_id` is set.

Database logic:
- Student statistics are derived from `enrollments`, `exam_attempts`, `exams`, `classes`, and `courses`.
- Completed scores use `attempt_questions` and `exam_questions` to calculate percentages.
- Class ranking uses a SQL ranking query partitioned by class code.
- Enrolled courses are joined through `classes` and `courses`.

The student module is gated by `user_logged_in()` to ensure only authenticated students can access these routes.

3.5. Exam-Taking Experience (Secure Proctoring)
------------------------------------------------

File: `testpoint/Student/templates/take_exam.html`

The exam interface implements a **secure lockdown mode** to prevent cheating and unauthorized access.

Lockdown and Security Features:
- **Fullscreen requirement**: Students must activate fullscreen mode before starting the exam. A lockdown overlay appears until fullscreen is activated.
- **Fullscreen exit detection**: If a student leaves fullscreen (via ESC or browser controls), a violation is logged and the overlay re-appears.
- **Tab switch detection**: Uses `visibilitychange` event to detect when students switch browser tabs or minimize the window.
- **Window blur detection**: Uses `blur` event to detect when students click outside the browser or focus another application.
- **Context menu disabled**: Right-click is prevented via `contextmenu` event handler.
- **Back button disabled**: History navigation is blocked via `history.pushState()` and `onpopstate` handler.
- **Violation throttling**: Violations are throttled to prevent spam logging (2-second cooldown between logs).
- **Instructor lockdown**: If a teacher restricts a student's access (403 response from server), a blocking modal appears with auto-logout countdown.
- **Unload prevention**: `onbeforeunload` warns students before leaving the page.

Exam UI Components:
- **Question display area**: Shows question text, difficulty badge (easy/medium/hard), and question type indicator.
- **Progress bar**: Visual indicator of answered questions as percentage of total.
- **Question counter**: Displays current question number and total question count.
- **Timer sidebar**: Shows remaining time in MM:SS format with warning color (red) when under 5 minutes.
- **Question map grid**: 5-column grid showing all questions with visual states:
  - Answered (green background, white checkmark implied).
  - Flagged for review (yellow background).
  - Not yet answered (white/light background).
- **Navigation buttons**: Previous Question, Next Question, Finalize & Submit.
- **Flag for Review button**: Toggles flag state for current question; highlighted when flagged.

Question Types:
- **Multiple choice**: Radio button selection with auto-shuffled options on page load.
- **True/False**: Radio button selection between True and False options.
- **Identification**: Text input field for typed short-answer responses.

Auto-Save and Progress Tracking:
- Answers auto-save to the server via fetch() AJAX calls when changed or navigated away.
- Current question index is tracked and saved with each progress update.
- Flag status (for review) is saved alongside the answer.
- `student.save_progress()` endpoint stores answer, flag state, and current index.
- `student.log_violation()` endpoint records tab switches, blur events, fullscreen exits, and other security violations.
- `student.submit_exam()` endpoint finalizes the exam, freezes answers, and prevents further modifications.

Violation Logging and Notification:
- Each security violation increments the `tab_switches` count in the `exam_attempts` table.
- Violations are sent to the server asynchronously without blocking the UI.
- A red violation banner appears at the top of the question area when violations are detected, with an alert icon.
- Violations are throttled to prevent multiple logs for the same event within 2 seconds.
- If a 403 status is returned from the server (instructor block), the exam freezes and triggers the blocked state.

Blocked/Restricted State (Instructor Action):
- When instructor restricts access, `triggerBlockedState()` is called immediately.
- Fullscreen is forced to exit to ensure modal visibility.
- All inputs (text, radio buttons, buttons) are disabled.
- The question map becomes unresponsive to clicks.
- A blocking modal with high z-index appears showing a security alert.
- The modal includes a countdown timer (starting at 5 seconds) before auto-logout.
- Page body is blurred using CSS filter to prevent further interaction.
- User is auto-logged out after countdown expires, redirecting to `/auth/logout`.

4. Teacher Module
-----------------

File: `testpoint/Teacher/teacher.py`

Teacher functionality includes:
- Teacher dashboard analytics:
  - number of distinct courses taught.
  - active examinees currently in-progress.
  - total questions in the teacher's bank.
  - total tab-switch violations across exams.
  - overall class average score.
  - question type distribution.
  - recent finished exam submissions.
- Question bank management:
  - view question banks by course.
  - view question details with options.
  - add questions in multiple-choice, true/false, and identification formats.
  - delete individual and bulk bank questions.
  - import questions from Excel files.
- Exam management:
  - manage exam draft creation and editing.
  - publish exams only after question selection.
  - copy/duplicate exams for reuse.
  - soft-archive exams with a trash bin.
  - permanently delete archived exams and related isolated questions.
- Question linking:
  - link bank questions to exams.
  - unlink questions from exams.
  - bulk link/unlink and perform bulk actions on exam questions.
- Enrollment and monitoring:
  - manage students enrolled in a class.
  - monitor live exam attempts.
  - view exam results and review specific student attempts.
  - reset student exam attempts if necessary.
- Profile page for teacher information and password updates.

Important details:
- Exams can be published to multiple classes and duplicated across class sections.
- The teacher module preserves class ownership and only allows modification by the creating teacher.
- The system distinguishes between banked questions and isolated exam-specific questions.
- `is_exam_locked()` prevents changes to active exams or exams with existing student submissions.

The teacher module is gated by `teacher_logged_in()` to protect teacher-only functionality.

5. Admin Module
---------------

File: `testpoint/Admin/admin.py`

Admin functionality includes:
- Overall dashboard metrics:
  - active user count.
  - active programs, blocks, and classes.
  - live exam sessions count.
  - global exam violation sum.
  - user role distribution.
  - student year distribution based on block names.
  - block capacity watchlist.
- Account management:
  - view active user records and related profile details.
  - edit account data and change passwords.
  - create admin, teacher, and student users.
  - soft-delete and restore accounts.
  - permanently delete users (restricted to `super_admin`).
- Program management:
  - create, edit, archive, restore, and permanently delete academic programs.
- Block management:
  - create blocks automatically by year level.
  - enforce capacity limits and prevent invalid resizing.
  - archive, restore, and permanently delete blocks.
  - manage current and unassigned students for each block.
  - assign and remove students from blocks, including bulk block assignments.
- Course management:
  - maintain a subject catalog of active and trashed courses.
  - add new courses and update subject metadata.
  - archive, restore, and permanently delete courses.
- Class management:
  - schedule classes linking course, block, and teacher.
  - auto-generate class codes like `#101`, `#102`.
  - prevent duplicate class assignments for the same course-block pair.
  - archive, restore, and permanently delete classes.
  - enroll entire blocks into classes.
- Enrollment management:
  - view and manage students enrolled in a class.
  - enroll and unenroll individual students.
- Exam oversight:
  - view all exams with teacher, course, block, and violation details.
  - see live exam status across the system.
- System logs and settings:
  - list user records and role-based styling for logs.
  - static admin settings page.
- Admin profile management.
- Verification queue:
  - view pending registration verifications and admin notes.
  - approve pending users to activate their accounts.
  - request resubmission of documents with reasons and notes for user correction.
  - reject registrations permanently with rejection categories and explanations.

The admin module is protected by `admin_logged_in()` and role-based controls.

6. Database and Data Model
--------------------------

The system is strongly database-driven and uses a schema named `test_point`.

Primary tables inferred from application code:
- `users`: active login accounts, password, role, verification, active flag, OTP metadata.
- `pending_users`: registration records awaiting OTP verification and document approval.
- `otp_table`: password reset OTP records with usage status.
- `students`, `teachers`, `admins`: user profile tables joined to `users`.
- `programs`: academic programs.
- `blocks`: year/section blocks with capacity and active flag.
- `courses`: course catalog entries.
- `classes`: scheduled class sections linking course, block, teacher.
- `enrollments`: student-to-class enrollments.
- `exams`: exam definitions, schedule, duration, pass percentage, active/inactive state.
- `exam_questions`: mapping between exams and question bank entries.
- `questions`: question bank entries and isolated exam questions.
- `options`: answer choices and correctness flags.
- `exam_attempts`: student exam sessions, scores, status, and violation counts.
- `attempt_questions`: record of questions served during an exam attempt.
- `student_answers`: submitted student answers for review.

Configuration values:
- Database host: `localhost`
- MySQL user: `root`
- Database name: `test_point`
- Authentication plugin: `mysql_native_password`
- Credentials loaded from `testpoint/passwordDB.env`

8. Security, UX, and Operational Notes
--------------------------------------

- Passwords are hashed using Werkzeug.
- Role-based access is enforced through explicit session checks in each blueprint.
- The app supports pending registration, email OTP verification, and admin approval.
- Reset password flow is OTP-based with usage and time limits.
- Uploaded verification documents are restricted to PDFs.
- No cache headers are set on responses to prevent stale session data in the browser.
- The admin namespace is isolated under `/admin`.
- Teachers and students are isolated under `/teacher` and `/student`, respectively.
- Exam-related session lockdown prevents students from leaving the exam flow during active attempts.
- Teachers can duplicate and publish exams to multiple classes.
- Admins can archive and restore programs, blocks, courses, classes, and users.

9. Observations for Developers
------------------------------

- The project is a single Flask application using blueprints for modularity.
- Business logic is implemented directly in route handlers with MySQL queries.
- There is no ORM layer; raw `mysql.connector` queries are used throughout.
- The code combines authentication, role management, academic scheduling, exam administration, and review workflows.
- Additional system state is managed via session variables across routes.
- The current README is minimal; the application file structure and blueprints carry most of the system design.
